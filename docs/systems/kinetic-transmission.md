# 动能传动系统（Transmission Shaft + Bevel Gear）

## 目标与范围

本规范定义基于以下方块的 KU（`KU/t`）传动：

- `wood_transmission_shaft`
- `iron_transmission_shaft`
- `steel_transmission_shaft`
- `carbon_transmission_shaft`
- `bevel_gear`

当前阶段只实现：

- 传动网络求解（服务端）
- KU 传播、衰减、承载上限
- 90° 伞齿轮换向
- 风力动能机作为 KU 源
- 动能发电机作为 KU 消费端（KU -> EU）

当前阶段不实现：

- 独立三向分动箱方块（暂不新增 `kinetic_t_junction`）
- 复杂变速比
- 过载破坏方块（先限幅）

## 单位与时间

- 动能单位：`KU/t`
- 网络每游戏刻（tick）求解一次（20Hz）
- 多源多汇在同一 tick 内统一参与求解，调度顺序对齐 `EnergyNetwork`
- KU 全程使用整数

## 节点与连接规则

所有传动块在网络层统一抽象为图节点。  
玩家搭建层面要求“有齿轮才可侧向耦合”，但求解层不模拟齿面接触几何。

### 传动轴（Shaft）

- 仅沿自身 `AXIS` 方向连接（正反两端）
- 与相邻方块连接判定：
- 相邻为传动轴：其 `AXIS` 必须等于连接方向轴
- 相邻为伞齿轮：其 `PLANE` 必须包含该连接方向轴

### 伞齿轮（Bevel Gear）

- `PLANE` 为 `XY` / `XZ` / `YZ`
- 仅在 `PLANE` 的两条轴方向上连接（各自正反，共 4 个方向）
- 用于 90° 换向传动
- 若机器想侧挂到轴上，必须通过伞齿轮接入（不允许直接贴轴侧面耦合）
- 传动模式约束（实现口径）：
- 允许：`2 面垂直（L 形）`
- 允许：`3 面 T 形（两面对穿 + 一面垂直）`
- 禁止：`3 面正交（X/Y/Z 各一面）`
- 禁止：`2 面面对面（仅对穿）`

## KU 传播算法（严格对齐 EnergyNetwork）

### 输入与求解策略

- 多源、多汇统一在同一张图里求解
- 网络内部不做“节点平均分流”或“伞齿轮均分”
- 每 tick 的调度顺序与 `EnergyNetwork` 一致：
- 先构建拓扑、边界端点和最短损耗路径
- 再由消费者按路径损耗从所有供给端拉取
- 路径被使用后，扣减路径上所有传动节点的本 tick 剩余吞吐

### 路径与损耗

- 对每个消费者，统计到所有供给端入口节点的候选路径
- 候选路径按总损耗升序尝试
- 路径总损耗 = 路径上所有传动节点的 `nodeLoss` 之和
- 路径可输送上限 = 路径上所有传动节点“本 tick 剩余容量”的最小值

### 单条路径结算

对一条候选路径，结算规则为：

1. 消费者先给出当前可接收的 `requestKu`
2. 供给端给出当前可输出的 `extractKu`
3. 该路径可交付的 KU 为 `min(requestKu, extractKu - pathLoss, pathCapacity - pathLoss)`
4. 真正经过路径的 KU 为 `insertedKu + pathLoss`
5. 路径上的每个传动节点都扣减这次经过的 KU，用于限制同 tick 其他路径复用

### 守恒

- 网络不会复制 KU
- 多个动能发生机串到同一网络时，发电机接收的 KU 等于所有可达供给端输出之和，再减去各自路径损耗
- 伞齿轮只是一个普通传动节点，不承担任何额外分配逻辑

## 承载上限与衰减参数

### 承载上限（KU/t）

- 木轴：`128`
- 铁轴：`512`
- 钢轴：`2048`
- 碳轴：`8192`
- 伞齿轮：`2048`

### 节点衰减（KU/t）

- 木轴：`0`
- 铁轴：`2`
- 钢轴：`1`
- 碳轴：`0`
- 伞齿轮：`3`

说明：

- 伞齿轮损耗高于直轴，体现换向损耗
- 过载先限幅，不破坏方块

## 多源与消费

### 多源与多汇

- 同一 tick 内多个源可同时向同一网络供给 KU
- 多个消费者共享同一组传动节点剩余吞吐
- 某条路径先被占用后，会直接减少后续路径可再利用的传输能力

### 消费端（当前只实现一种）

- `kinetic_generator`：将 KU 转为 EU（本阶段唯一 KU 消费机器）
- 固定转换率：`4 KU = 1 EU`
- 每 tick 最大吞吐：`512 EU/t`（等价 `2048 KU/t`）
- EU 缓冲：`10000 EU`
- 启停阈值：`64 KU/t` 启动，`48 KU/t` 停止（滞回，避免临界抖动）
- 后续新增动能机仅需实现 Source 或 Sink 接口，不改网络核心

## 风力动能机接入

- 风力动能机作为 KU 源
- 当转子存在且未卡住时注入 KU
- 动能输出口在机器背面（正面为叶片渲染面）
- 注入入口：方块朝向背面的相邻方块（`pos.offset(facing.opposite)`）

### 转子等级与倍率

- 当前实现等级：木 / 青铜 / 铁 / 钢 / 碳
- 等级倍率：`1 / 2 / 2 / 3 / 4`

### 转子寿命

- 转子为有耐久物品，拆下后寿命会保留
- 只有在风机实际转动并产生 KU 时才会损耗寿命
- 基准损耗速度按晴天 `1.0x` 计算；天气倍率会等比放大寿命损耗
- 当前寿命：
- 木转子：`3` 小时
- 青铜转子：`24` 小时
- 铁转子：`24` 小时
- 钢转子：`48` 小时
- 碳转子：`168` 小时
- UI / Jade / 物品 tooltip 显示的剩余寿命口径为“晴天标准剩余寿命”，不随当前天气倍率抖动

### 阻挡与卡住

- 若叶片前方有实体方块，会按叶片采样路径判定阻挡
- 若两台风力动能机转子处于同一旋转平面，且两个圆盘发生重叠，也会直接判定卡住
- 卡住时机器停止输出 KU，叶片固定在预设卡住角度显示
- Jade 与机器 Screen 会显示当前阻挡状态

### 风强模型（仅基于 Y）

- 不考虑相对地面高度，只使用绝对 `y`
- 平均风强使用高斯分布，峰值在 `y = 150`
- 当前参数：
- `mu = 150`
- `sigma = 35`
- `floor = 0.03`（地面附近接近 0 但不为 0）
- 公式（归一化均值）：
- `mean(y) = floor + (1 - floor) * exp(-((y - mu)^2) / (2 * sigma^2))`

### 时间扰动与天气倍率

- 每 10 秒（200 tick）更新一次风扰动因子
- 扰动采用“按区块缓存”的随机风：
- 每个区块维护自己的 `gustFactor`
- 每个区块有固定的更新时间偏移，更新时刻会均匀分摊到 200 tick 窗口内
- 同一区块内的风机共享同一个 `gustFactor`
- 扰动范围：`[0.5, 1.5]`（即上下浮动 ±50%）
- 天气倍率：
- 晴天：`1.0`
- 下雨：`1.2`
- 雷暴：`1.5`

### 输出与启动阈值

- 有效风强：`effectiveWind = mean(y) * gustFactor * weatherFactor`
- KU 输出：`kuOut = baseKu * rotorMultiplier * effectiveWind`
- 每级最小启动风力阈值按同倍率关系放大（木/铁/钢/碳 = `1/2/3/4`）：
- 木：`Wmin * 1`
- 铁：`Wmin * 2`
- 钢：`Wmin * 3`
- 碳：`Wmin * 4`
- 该约束保证钢/碳仅在高空稳定工作

## 搭建规则（面向玩家）

- 轴与轴可直连（同轴向）
- 轴与机器的侧向耦合需要伞齿轮作为中间节点
- 网络层统一按传动节点连接关系求解，不模拟齿面几何细节

## 渲染与逻辑边界

- 传动求解仅在服务端执行
- BER 渲染转速与当前传递 KU 成正比（分段线性映射）
- 当前渲染口径不是“每节点各自独立 through-put”，而是“整张连通动能网统一渲染 KU”
- 服务端会把这一个统一渲染 KU 写入该网络内所有传动节点的 `currentKu`
- 只有当整网渲染 KU 发生变化时，才对这些节点做脏同步；稳态下不会每 tick 清零再全网重发

### 转速映射（当前约定）

- 设 `ku = currentKu`
- 分段：
- `0..512 KU`：斜率 `k1`
- `512..2048 KU`：斜率 `k2 = 0.5 * k1`
- `2048..8192 KU`：斜率 `k3 = 0.25 * k1`
- 最终 `rpm = min(rpmMax, piecewise(ku))`

## 标定点（强制约束）

- 条件：
- 碳转子
- `y = 150`
- 晴天（天气倍率 `1.0`）
- 无随机扰动（`gustFactor = 1.0`）
- 目标输出：`128 EU/t`
- 由 `4 KU = 1 EU` 可得该标定点 KU 输出应为：`512 KU/t`
- 因碳转子倍率为 `4`，该条件下基础风强输出基准应满足：
- `baseKuAtPeak = 128 KU/t`（木转子峰值基准）
