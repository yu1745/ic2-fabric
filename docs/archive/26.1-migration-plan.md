# 26.1 迁移计划（搁置，待资源就绪后再推进）

## 问题

### 背景

1. `main` (1.20.1) + `1.21.1` 分支使用 **Yarn 映射**
2. 26.1 是非混淆版本，必须使用 **Mojang 官方映射**
3. 项目约 570+ 个 Kotlin 源文件，代码中大量使用 Yarn 命名的类/方法

### 迁移难点

直接升级 26.1 时有约 **20,800+ 编译错误**，分两类：

| 类别 | 占比 | 示例 |
|------|------|------|
| **A. 命名差异**（Yarn→Mojang）| ~30% | `World`→`Level`、`ScreenHandler`→`AbstractContainerMenu`、`NbtCompound`→`CompoundTag` |
| **B. API 差异**（Mojang 1.21→26.1 自身变化）| ~70% | 方法签名变化、类重组、Fabric API 重构 |

两类错误交织在一起，难以批量处理。

### 分支策略

- `main` (1.20.1 Yarn) = 核心开发分支，**不动**
- `1.21.1` (Yarn) = cherry-pick 中继，**不动**
- `26.1` (Mojang) = 独立迁移，无法从上游直接 cherry-pick

这意味着每次从 `main` cherry-pick 到 `26.1` 都需要手动处理命名转换。

## 方案：利用重叠版本自动提取 Yarn→Mojang 映射

### 思路

MC 1.21.5 ~ 1.21.11 期间，Loom 同时支持 Yarn 和 `loom.officialMojangMappings()`。在此期间 **两套命名并存**，可以通过对比 mapping 文件自动提取完整的 Yarn→Mojang 映射表（包括类名、方法签名、字段名）并生成批量转换脚本。

### 实现步骤（待做）

1. 在 1.21.x 版本搭建一个临时构建，同时加载 Yarn 和 Mojang mapping
2. 使用 `mapping-io` 读取两者，差分出完整的 `Yarn→Mojang` 翻译表
3. 将翻译表落地为 `.tiny` 或 `migration-map.json`
4. 编写转换脚本，将 1.20.1/1.21.1 的源码直接翻译为 Mojang 命名版本
5. 在 `26.1` 分支应用，首先消除**全部 A 类错误**
6. 剩余的 B 类错误即为纯 26.1 API 变更，逐个修复即可

### 预期效果

- 一次提取，全量覆盖（类、方法、字段）
- 避免手写几百条正则规则、漏改误改
- B 类错误独立出来后可针对性修复

## 当前状态

- `mc_source_26.1` 已生成（Mojang 命名，4789 common + 2093 client 文件）
- `fabric-api_26.1` 已从 26.1.2 分支检出
- 已完成约 571 文件的批量重命名（手工规则，非全量）
- 剩余 ~20,846 编译错误
